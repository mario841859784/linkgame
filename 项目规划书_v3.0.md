## 1. 概述与产品定位

本项目是一款基于 Flutter 开发的 Android 端经典消除类游戏。玩家需在网格棋盘中，通过转弯不超过 2 次的连线路径，将两个相同 Emoji 图标配对消除。

### 1.1 目标群体

产品核心受众定位于**老年群体及视觉障碍用户（视力退化、白内障患者）**。系统设计全面贯彻适老化与无障碍原则：

* **视觉防遮挡**：指尖悬浮放大镜，手指操作时不遮挡目标格子
* **大尺寸交互域**：网格自适应充满屏幕，22sp-56sp 超大字号
* **多模态反馈**：WAV 音效 + 震动 + TTS 语音播报
* **防卡关**：15 秒无操作自动提示，无解时自动洗牌

## 3. 核心机制

### 3.1 寻路算法
* **0 转弯**：两点间直线，无障碍物
* **1 转弯**：单直角拐点
* **2 转弯**：双拐点，扫描平行线段

### 3.2 控制流
* **死局检测**：遍历所有未消除同类格子，找到配对即返回
* **自动洗牌**：Fisher-Yates 乱序剩余 Emoji
* **提示系统**：复用死局检测逻辑，返回首个配对坐标

* 竖屏自适应：`min(宽度/列, 高度/行)`

## 4. 多模态交互设计 (Phase 1)

针对视觉障碍老年用户，单次精准点击难度极高。用户按住起点格子滑动至终点释放，全程叠加四重补偿：

### 4.1 视觉补偿 — 指尖悬浮放大镜

手指按住滑动时，在触摸点正上方（Y 偏移 -80dp）悬浮渲染当前格子的大号 Emoji + 白底黑框，解决"手挡屏幕"问题。

* 技术：`Stack` + `Positioned`，监听 `onPanUpdate` 动态更新坐标
* 触发：手指按下并移动超过 10dp 后显示，松手隐藏

### 4.2 触觉补偿 — 网格边界感知震动

手指滑动跨越不同格子时触发极轻微震动（15ms），提供"拨动齿轮"的段落感。

* 维护 `lastHoveredIndex`，索引变化时触发
* 匹配成功：50ms 强震
* 匹配失败：30ms + 30ms 间隔震动

### 4.3 听觉补偿 — TTS 驻留播报

手指滑入某格子并停留超过 0.2 秒，TTS 播报该图案中文名称。

* 需节流防抖：避免高频播报导致卡顿
* 匹配成功时额外播报"配对成功"
* 支持设置页 TTS 开关

### 4.4 行为容错 — 首尾焦点判定

用户只需"按住 A，滑到 B，松手"。系统取起点和终点坐标做路径计算，不要求轨迹精确。

* 滑动阈值：移动 ≥ 30dp 才识别为滑动，否则视为点击
* 使用 `GestureDetector.onPan` 管理手势生命周期

## 5. Phase 1 执行步骤

### 5.1 数据结构重构

**目标**：Cell 增加 TTS 标签，Emoji 主题绑定中文映射

* `Cell` 模型增加 `ttsLabel` 字段 (String)
* `EmojiTheme` 增加 `ttsLabels` 字段 (List<String>)，与 emojis 一一对应
* `GameState` 初始化 Cell 时同时赋值 `ttsLabel`

**涉及文件**：`cell.dart`, `emoji_themes.dart`, `game_state.dart`

### 5.2 引入 flutter_tts

**目标**：TTS 引擎初始化，封装播报工具

* `pubspec.yaml` 添加 `flutter_tts` 依赖
* 创建 `lib/utils/tts_service.dart` 单例：
  - `init()`：设置语言为 zh-CN，设置语速、音调
  - `speak(String text)`：播报文本，支持节流（距上次播报 < 0.2s 时跳过）
  - `stop()`：停止当前播报
  - `enabled` 开关（从 Settings 读取）

**涉及文件**：`pubspec.yaml`, `tts_service.dart`, `settings_service.dart`

### 5.3 手势系统迁移

**目标**：`Listener` → `GestureDetector`，统一 `onPan` 手势

* 用 `GestureDetector` 的 `onPanDown` / `onPanUpdate` / `onPanEnd` 替换当前 `Listener` 的 `onPointerDown` / `onPointerMove` / `onPointerUp`
* `onPanDown`：记录起点格子坐标，缓存 RenderBox
* `onPanUpdate`：更新当前悬浮格子坐标，触发跨界震动和 TTS 驻留计时
* `onPanEnd`：首尾判定，调用 `tryMatch()` 或直接 fallback 到 `_onCellTap()`

**涉及文件**：`play_session_screen.dart`

### 5.4 指尖悬浮放大镜 Widget

**目标**：随手指移动的大号 Emoji 悬浮视图

* 新建 `lib/widgets/magnifier_widget.dart`
* 属性：`emoji`(String), `position`(Offset), `cellSize`(double)
* 渲染：直径 = cellSize * 1.5 的圆形白底黑框，中心显示超大 Emoji（字号 = cellSize * 0.8）
* 在 `PlaySessionScreen` 的 `Stack` 中根据 `isDragging` 状态条件渲染

**涉及文件**：新建 `magnifier_widget.dart`, `play_session_screen.dart`

### 5.5 跨界感知震动 + TTS 驻留播报

**目标**：滑动过程中的实时反馈

* 在 `onPanUpdate` 中：
  - 计算当前手指所在格子索引
  - 与 `lastHoveredIndex` 对比，不同则触发 15ms 微震并更新
  - 启动/重置 0.2s 驻留计时器，到期后调用 TTS 播报当前格子名称
* 在 `onPanEnd` 中：清除驻留计时器

**涉及文件**：`play_session_screen.dart`, `tts_service.dart`

### 5.6 设置页扩展

**目标**：增加 TTS 语音播报开关

* `SettingsScreen` 增加 Switch 控件："语音播报"
* `SettingsService` 增加 `isTtsEnabled` 读写
* 默认开启

**涉及文件**：`settings_screen.dart`, `settings_service.dart`

### 5.7 App 生命周期管理

**目标**：切后台暂停计时和 TTS，切回前台恢复

* 监听 `AppLifecycleState`：
  - `paused`：暂停 `_idleTimer`，调用 `TtsService.stop()`
  - `resumed`：重置 `_idleTimer`
* 复用已有的 `app_lifecycle.dart`

**涉及文件**：`app_lifecycle.dart`, `play_session_screen.dart`

## 8. 多 Agent 协作编排

按技术架构拆分为 5 个能力域，映射 OpenCode 现有 Agent，支持并行执行。

### 8.1 能力域与 Agent 映射

| 能力域 | Agent | 职责 |
|---|---|---|
| UI/UX 设计与前端实现 | `designer` | 放大镜 Widget、设置页、游戏页布局、胜利页动效、适老化视觉 |
| 游戏核心逻辑 | `fixer` | Cell/Emoji 模型、GameState、PathFinder 寻路、死局检测、提示系统、计分 |
| 多模态反馈集成 | `fixer` | TTS 服务封装、音效播放、震动反馈、flutter_tts 依赖 |
| 手势系统与生命周期 | `fixer` | GestureDetector 迁移、跨界震动、TTS 驻留计时、AppLifecycleState |
| 单元测试 | `tdd-guide` | path_finder 和 game_state 单元测试，80%+ 覆盖率 |

### 8.2 执行顺序与依赖

```
orchestrator (总调度)
│
├── Phase 1A: fixer (并行执行，无依赖冲突)
│   ├── 域2: 游戏核心逻辑 (Cell/Emoji/GameState/PathFinder)
│   ├── 域3: 多模态反馈 (TTS/Audio/Haptic)
│   └── 域4: 手势系统 + 生命周期
│
├── Phase 1B: designer (依赖 1A 完成的数据模型和服务)
│   └── 域1: UI 实现 (放大镜/设置页/游戏页/胜利页)
│
└── Phase 2: tdd-guide (依赖 1B 完成)
    └── 域5: 单元测试 (path_finder + game_state)
```

### 8.4 Agent 协作规则

1. **fixer 三域并行**：域2/3/4 无文件冲突，可同时执行
2. **designer 串行等待**：必须等 fixer 完成数据模型后再实现 UI
3. **tdd-guide 最后执行**：依赖所有功能代码完成后编写测试
4. **code-reviewer 全程介入**：每个域完成后自动 review
5. **orchestrator 统一调度**：负责依赖检查、进度跟踪、冲突解决
